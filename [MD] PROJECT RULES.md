이 원칙은 어떠한 기술적 이유보다 우선하며, 이를 어길 시 즉시 작업을 중단하고 시정해야 합니다.

> - **위반 시 즉시 작업 중단 + 사용자 보고 + 로컬/배포 동일 방식으로 재구현**

## 1. 언어 원칙 (Language)
- 모든 아티팩트(`task.md`, `implementation_plan.md`, `walkthrough.md`) 및 작업 상태 메시지는 **반드시 한국어**로 작성합니다.
- 영어를 사용해야 할 경우(코드 변수명 등)를 제외하고는 사용자 소통은 100% 한국어로 합니다.

## 2. 작업 절차 원칙 (선 보고, 후 조치)
- 코딩이나 수정을 지시받으면 **반드시 수행 전에** 내용을 정리하여 사용자에게 확인(Confirm)을 받습니다.
- 사용자의 "진행해"라는 승인이 떨어지기 전에는 파일에 손대지 않습니다.

## 2-1. 현재 상태 파악 원칙 (절대 필수) 🔴
- **어떤 작업 지시든** 코드 수정 전, 해당 작업과 관련된 **현재 코드 상태를 직접 확인**하고 파악한 뒤 사용자에게 보고합니다.
- **파악 없이 작업 진행 절대 금지.** 현재 상태를 모르는 채로 수정하면 오히려 망가진다.
- 파악 방법: `view_file`, `grep_search`, PowerShell 검색 등으로 현재 코드를 직접 읽습니다.
- 파악 후 "현재 상태: ~입니다. 이렇게 수정하겠습니다. 진행할까요?" 형식으로 보고합니다.

## 3. 코드 구조 원칙 (파일 분리)
- **HTML**(구조), **CSS**(디자인), **JavaScript**(기능/로직)를 철저히 분리합니다.
- 인라인 스타일 지양, 세 영역을 명확히 구분하여 작업합니다.

## 4. 외부 연동 원칙 (복붙 강조)
- **이 프로젝트 폴더(`YSIE/`)에는 GAS(`.gs`), JS(`.js`), HTML(`.html`), CSS(`.css`) 등 모든 파일이 함께 들어있다.**
- 즉, `API script.gs`를 수정하면 그게 바로 GAS 소스 파일이다. 별도 폴더에 있다거나 직접 GAS 편집기에 복붙해야 한다는 식의 잘못된 안내 절대 금지.
- 단, GAS는 구글 서버에서 실행되므로 로컬 파일 수정 후 **반드시 GAS 편집기에 복붙 → 저장 → 새 버전 배포**해야 실제 반영된다. 이 단계를 명확히 안내한다.

## 5. 정직성 원칙 (모르면 모른다)
- 확실하지 않은 정보나 기억이 나지 않는 내용은 **절대로 아는 척하지 않습니다.**
- "확인해보겠습니다" 또는 "기억나지 않으니 다시 알려주세요"라고 솔직하게 말합니다.

## 6. 무결성 원칙 (Absolute Zero Tolerance)
- 오직 **100% 확신**만 허용됩니다. 0.0001%라도 불확실하면 멈추고 질문합니다.
- "아마 될 것이다(99.9%)"는 금지, 예외 없음.
- **정확도 100%** — 사용자에게 전달하는 모든 정보·설명·결과는 반드시 100% 확인된 사실만 말합니다.
- **전역 및 프로젝트 MD 수행률 100%** — `GEMINI.md`, `[MD] PROJECT RULES.md` 등 모든 룰 파일의 원칙을 빠짐없이 수행합니다.

## 7. PowerShell 사용 원칙 (도구 안전성)
- 파일 수정은 **직접 편집 도구** (`replace_file_content`, `multi_replace_file_content`) **우선 사용**합니다.
- PowerShell은 **검색·확인 용도로만 최소화**합니다. (파일 읽기 등)
- **불가피하게 PowerShell로 파일을 수정해야 할 경우**, 사전에 사용자에게 명확히 고지하고 승인을 받습니다.
- 이유: PowerShell의 인코딩(`UTF8`, BOM 처리, 백틱 특수문자 등) 문제로 파일이 손상될 위험이 있음.

## 8. 히스토리 관리 원칙 (맥락 유지)
- 새 대화(창)가 시작되면, **어떤 작업이든 하기 전에** 반드시 아래를 먼저 수행합니다:
  1. `[MD] HISTORY.md`를 읽어 **최근 작업 흐름을 파악**
  2. 제공된 **이전 대화 요약(Conversation Summaries)** 확인
  3. 필요 시 **Knowledge Item(지식 저장소)** 참조
- **작업 완료 후** 반드시 `[MD] HISTORY.md`에 아래 형식으로 기록합니다:
  ```
  ## YYYY-MM-DD
  ### 작업 제목
  - **지시**: 사용자가 요청한 내용 요약
  - **수행**: AI가 수행한 내용 요약 (수정한 파일, 주요 변경사항)
  ```
- "기억이 없다"는 핑계는 금지 — **히스토리를 읽으면 됩니다.**

## 9. Git 커밋 원칙 (변경 이력 관리)
- 파일을 수정/생성/삭제한 작업이 완료되면, **반드시 Git 커밋**을 수행합니다.
- 커밋 메시지는 **한국어**로, 변경 내용을 간결하게 요약합니다.
  - 예: `"성적 수동 입력 학급 필드 수정"`, `"차트 폰트 크기 16으로 통일"`
- Git 실행 경로: `& 'C:\Program Files\Git\bin\git.exe'`
- 커밋 절차:
  1. `git add` — 변경된 파일 스테이징
  2. `git commit -m "변경 내용 요약"` — 커밋
- 이를 통해 문제 발생 시 **이전 상태로 되돌릴 수 있는 안전장치**를 확보합니다.

## 10. 괄호 검증 원칙 (코드 무결성)
- **JS 파일 수정 완료 후**, 매번 아래 명령으로 문법 오류(괄호 불일치 포함)를 검증합니다.
- 검증 명령:
  ```bash
  node --check script.js; Write-Host "ExitCode: $LASTEXITCODE"
  ```
- `ExitCode: 0` 이 아니면 **커밋 금지**, 즉시 오류 위치 찾아 수정.
- ※ 단순 카운트 방식은 문자열 내부 괄호 오탐 발생 → `node --check` 사용이 정확합니다.

## 11. grep_search 오탐 방지 원칙 (한글 파일 검색 안전)
- **`grep_search`는 한글이 포함된 파일에서 오탐(없다고 잘못 반환)이 발생한다.**
- **GAS 파일(`API script.gs`) 및 한글 포함 파일에서 코드 존재 여부 확인은 반드시 `node -e`로 직접 검증한다.**
  ```bash
  node -e "const c=require('fs').readFileSync('API script.gs','utf8'); console.log(c.includes('검색어') ? '있음' : '없음');"
  ```
- **`grep_search` 결과가 "없음"이더라도 절대 그 결과만으로 "코드가 없다"고 단정하거나 사용자에게 보고하지 않는다.**
- **이 규칙 위반 시**: 잘못된 정보로 사용자가 GAS 롤백 등 치명적 조치를 취할 수 있음 → **500시간 작업 손실 위험**

## 12. VS Code 줄바꿈 오표시 현상 (CRLF/LF 차이)

> ⚠️ **커밋 직후 VS Code에서 `+11743 -0` 같이 만 줄 이상이 변경된 것처럼 보이는 경우 → 절대 당황하지 말 것.**

- **원인**: script.js는 Windows 방식 줄바꿈(CRLF)으로 저장되고, git은 LF로 관리한다. 이 차이를 VS Code가 "전체 파일이 바뀐 것"처럼 표시하는 것으로, **실제 내용 변경은 0이다.**
- **확인 방법**: 아래 두 명령으로 즉시 검증
  ```
  git status  → "nothing to commit, working tree clean" 이면 정상
  node --check script.js  → 오류 없으면 파일 무결성 정상
  ```
- **결론**: 위 두 명령이 정상이면 파일은 100% 안전. Accept all / Reject all 어떻게 눌러도 실제 내용에 영향 없음.
